## 📠 如何把「本單位所有印表機資訊」整理、呈現，打造最能輔助管理的視覺化 Dashboard  

以下內容分為四大段落，從 **資料蒐集 → 資料模型 → 主要指標 (KPI) → 可視化與報表**，一步步說明如何把印表機資訊變成管理層可直接判斷、快速決策的資訊。  
所有範例皆以 **繁體中文** 撰寫，並提供可直接套用的工具與實作建議。

---

## 1️⃣ 資料蒐集與整理

| 來源 | 取得方式 | 建議工具 |
|------|----------|----------|
| **印表機硬體資訊** (型號、序號、IP、MAC、安裝位置、購買日期、保固結束日) | SNMP、WMI、REST API、或廠商提供的管理介面 (HP iLO, Canon iR‑ADV, Epson Net Config 等) | - `snmpwalk` / `snmpget` <br>- PowerShell (`Get‑Printer`, `Get‑CimInstance`) <br>- Python `pysnmp`、`netmiko` <br>- 商用工具：SolarWinds NPM、PRTG、ManageEngine OpManager |
| **耗材狀態** (碳粉/墨水余量、紙匣容量、警示門檻) | SNMP OID、印表機內建的 Web API、或透過 CUPS (Linux) 取得 | - `snmpwalk -v2c -c public <IP> .1.3.6.1.2.1.43` <br>- CUPS `lpstat -p -l` |
| **使用量統計** (列印頁數、列印時間、使用者、文件類型) | 印表機內部日誌、伺服器端列印佇列 (Windows Print Server, CUPS)、或第三方列印管理平台 (PaperCut, Printix) | - Windows Event Log + PowerShell <br>- CUPS `lpstat -W completed` <br>- PaperCut API |
| **故障與維修紀錄** (錯誤碼、停機時間、維修工單、維修成本) | ITSM 系統 (ServiceNow、Jira Service Management) 或手動 CSV 匯入 | - API 抽取或 CSV 匯入 |
| **網路與安全資訊** (IP、子網、VLAN、TLS/HTTPS 設定、認證方式) | 網路掃描、MDM、或印表機自帶的安全報表 | - Nmap、Nessus、Qualys |

> **小技巧**：  
> - 建議先在 **測試環境** 用腳本跑一次全域掃描，確定所有 OID/API 正確。  
> - 為避免 SNMP 被防火牆封鎖，可在網段內部署 **代理伺服器** (例如使用 `snmpcollector` Docker) 先收集再傳至中心資料庫。

## 2️⃣ 資料模型 (Database Schema)

| Table | 主要欄位 | 說明 |
|-------|----------|------|
| `printers` | `printer_id` (PK, UUID) <br> `model` <br> `serial_number` <br> `ip_address` <br> `mac_address` <br> `location` <br> `purchase_date` <br> `warranty_end` <br> `manufacturer` | 基本硬體資訊 |
| `supplies` | `supply_id` (PK) <br> `printer_id` (FK) <br> `type` (toner/ink/paper) <br> `capacity` <br> `remaining_percent` <br> `last_refill_date` | 耗材狀態 |
| `print_jobs` | `job_id` (PK) <br> `printer_id` (FK) <br> `user_id` <br> `pages` <br> `color_pages` <br> `black_white_pages` <br> `timestamp` <br> `document_name` | 每筆列印紀錄 |
| `incidents` | `incident_id` (PK) <br> `printer_id` (FK) <br> `error_code` <br> `description` <br> `start_time` <br> `end_time` <br> `downtime_minutes` <br> `resolution` <br> `cost` | 故障與維修 |
| `users` | `user_id` (PK) <br> `name` <br> `department` <br> `email` | 列印使用者 (可與 AD 同步) |
| `printer_network` | `printer_id` (FK) <br> `vlan` <br> `subnet` <br> `dns_name` <br> `tls_enabled` | 網路安全設定 |

> **實作建議**：  
> - 使用 **PostgreSQL** + **TimescaleDB**（時間序列）儲存 `print_jobs`，方便做大數據分析。  
> - 若想快速部署，可考慮 **Supabase**（自帶 Postgres、Auth、Realtime）或 **Google Firestore**（NoSQL）+ **Cloud Functions** 收集資料。

## 3️⃣ 關鍵指標 (KPI) – 管理層最關心的資訊

| KPI | 計算方式 | 為什麼重要 | 建議呈現方式 |
|-----|----------|------------|--------------|
| **印表機利用率** (列印頁數 ÷ 設備可用天數) | `SUM(pages) / DATEDIFF(CURRENT_DATE, purchase_date)` | 判斷哪些機器使用頻繁、是否需要升級 | 折線圖 / 熱力圖 |
| **耗材耗盡預警** | `remaining_percent < 15%` | 防止印表機在關鍵時刻缺墨/碳粉 | 儀表板紅色警示、Email/Slack 通知 |
| **故障率** (故障次數 / 1000頁) | `COUNT(incidents) / SUM(pages) * 1000` | 了解機器可靠度，提前更換高故障率機型 | 條形圖 (部門或機型) |
| **平均停機時間** | `AVG(downtime_minutes)` | 直接影響部門工作效率 | KPI 卡片 + 趨勢線 |
| **每部門列印成本** | `(toner_cost + paper_cost) * pages / 1000` | 成本控制、預算分配 | 圓餅圖或堆疊條形圖 |
| **碳排放量** (估算) | `pages * 0.005 kg CO₂`（依廠商公佈） | ESG 報告、綠色辦公 | 數值卡 + 累積圖 |
| **保固剩餘天數** | `DATEDIFF(warranty_end, CURRENT_DATE)` | 預防保固過期、安排維修合約 | 列表 + 近 30 天紅色標示 |
| **紙張使用量** | `SUM(paper_tray_remaining)` 或 `pages` | 省紙、環保政策 | 趨勢圖 + 目標線 |
| **列印安全性** (TLS/HTTPS 啟用比例) | `COUNT(CASE WHEN tls_enabled THEN 1 END) / COUNT(*)` | 防止資料外洩 | KPI 卡片 + 警示 |

> **可視化建議**：  
> - **儀表板**：使用 **Grafana**, **Power BI**, **Tableau** 或 **Superset**。  
> - **即時警示**：Grafana + **Alertmanager**，或 **Microsoft Teams/Slack** Webhook。  
> - **行動版**：將關鍵 KPI（利用率、耗材警示、故障率）做成手機 Dashboard，讓 IT 主管隨時掌握。

## 4️⃣ Dashboard 範例設計（以 Grafana 為例）

### 4.1 整體布局

| 區塊 | 內容 | 圖表類型 |
|------|------|----------|
| **左上角** | **總體概況卡片**<br>• 印表機總數<br>• 已啟用 TLS %<br>• 近 30 天故障次數 | KPI 卡片 (Stat) |
| **左中** | **耗材預警表**<br>列出 `remaining_percent < 10%` 的耗材 | Table (帶超連結至維護工單) |
| **左下** | **保固即將到期** | Table + 紅色標示 |
| **中間上** | **印表機利用率熱力圖**（列印頁數 / 天） | Heatmap (時間 vs 印表機) |
| **中間下** | **部門列印成本** | 堆疊條形圖 (部門) |
| **右上** | **故障率 & 平均停機時間** | 雙軸折線圖（故障率、停機時間） |
| **右中** | **耗材消耗趨勢**（碳粉、墨水） | Area Chart |
| **右下** | **碳排放量累積** | 累積條形圖 + 目標線（公司減碳目標） |

> **視覺化小技巧**  
> 1. **顏色規則**：綠 (正常) → 黃 (警告) → 紅 (危急)。  
> 2. **互動**：點擊熱力圖的某台印表機，可跳出 **詳細頁**（顯示最近 30 天的列印趨勢、耗材更換紀錄、故障紀錄）。  
> 3. **時間範圍**：提供「今日 / 本週 / 本月 / 自訂」快速切換。

### 4.2 報表與匯出

| 報表 | 目的 | 匯出格式 |
|------|------|----------|
| **月度列印成本報表** | 部門預算審核 | PDF、Excel |
| **耗材採購建議** | 供應鏈規劃 | CSV |
| **故障與維修統計** | IT 維運評估 | PDF、PowerPoint (圖表) |
| **ESG 碳排放報告** | 企業永續報告 | PDF、JSON (供外部系統上傳) |

> **自動化**：使用 **Grafana Reporting** 或 **Power Automate**，每月第一天自動產生 PDF 並寄給財務與管理層。

## 5️⃣ 實作流程概覽（從零到可用）

1. **規劃需求**  
   - 與管理層確認最關心的 KPI（例如「耗材警示」與「部門列印成本」）。  
   - 決定資料更新頻率（即時、每 5 分鐘、每日一次）。

2. **建置資料收集腳本**  
   - 使用 **Python** 撰寫 SNMP 讀取腳本，排程 (`cron` / `Windows Task Scheduler`) 每 5 分鐘抓一次耗材與狀態。  
   - 使用 **PowerShell** 收集 Windows Print Server 列印紀錄，寫入同一資料庫。  
   - 若已有 **PaperCut** / **Printix**，直接呼叫他們的 REST API。

3. **建立資料庫**  
   - 建立上述的 `printers`、`supplies`、`print_jobs` 等表格。  
   - 設定 **索引**：`printer_id`, `timestamp`, `user_id`，提升查詢效能。

4. **資料清洗與 ETL**  
   - 用 **dbt**、**Airflow** 或 **Apache NiFi** 轉換原始 SNMP 數值為「剩餘百分比」等友善欄位。  
   - 每日跑一次 **聚合表**（如 `printer_daily_summary`），存放每日列印總頁數、耗材消耗量，供 Dashboard 快速查詢。

5. **建置 Dashboard**  
   - 在 **Grafana** 新增資料來源 (PostgreSQL)。  
   - 依前面示範的布局建立 Panel，設定 **Alert**（如耗材 < 15%）。  
   - 加入 **User Authentication**（OAuth2 / LDAP）確保只有授權人員能看到。

6. **自動化報表**  
   - 使用 **Grafana Reporting** 或 **Power BI** 的排程功能，將重要報表自動寄給部門主管。  
   - 若公司使用 **Microsoft Teams**，可以直接把圖表嵌入 Teams Tab。

7. **持續改進**  
   - 每季收集使用者回饋（管理層、IT、使用者），調整 KPI 或圖表。  
   - 加入 **機器學習**：預測耗材更換時間、故障風險（使用 `scikit‑learn` 或 `Azure ML`），未來可在 Dashboard 中顯示「預測維修日期」。

## 6️⃣ 常見問題 & 建議解決方案

| 問題 | 可能原因 | 解決方案 |
|------|----------|----------|
| **印表機未被偵測到** | SNMP 被防火牆阻擋、社群字串錯誤、IP 位址變更 | 確認防火牆允許 UDP 161/162；使用 `nmap -sU -p 161 <subnet>` 確認可達；將 IP 與 MAC 記錄在 `printer_inventory` 表，若 IP 變動自動更新。 |
| **耗材數值不正確** | 不同品牌 OID 不同、單位不同（% vs 紙張數） | 建立 **品牌對應表**（brand → OID），在收集腳本中根據 `manufacturer` 轉換為統一的 `%`。 |
| **列印紀錄量太大** | 高流量環境（每日上萬筆） | 使用 **TimescaleDB** 的分區表或 **ClickHouse**，僅保留 12 個月的原始資料，過期資料匯總至 `monthly_summary`。 |
| **警示太多** | 設定門檻過低、重複通知 | 在 Alert 中加入 **抑制規則**（同一台印表機在 1 小時內只發一次），並使用 **Severity** 分級。 |
| **管理層不懂技術圖表** | 使用過於複雜的圖形 | 為每個 KPI 加上 **文字說明**（Tooltip），或提供 **一頁摘要**（Executive Summary）只顯示關鍵數值與趨勢箭頭。 |

## 7️⃣ 推薦工具與範本

| 功能 | 推薦工具 | 為什麼適合 |
|------|----------|------------|
| **SNMP 收集** | `snmpcollector` (Docker) + `Grafana Loki` | 容器化、可擴充、即時推送到 Loki 再寫入 DB |
| **列印紀錄** | **PaperCut MF** (內建 API) | 完整列印審計、使用者對應 AD |
| **資料庫** | **PostgreSQL + TimescaleDB** | 時間序列查詢快、支援聚合 |
| **Dashboard** | **Grafana** (開源) + **Grafana Alerting** | 豐富圖表、易於自訂、支援多種資料源 |
| **報表自動化** | **Power Automate** 或 **Grafana Reporting** | 可排程產 PDF、直接寄信 |
| **通知** | **Slack / Microsoft Teams Webhook** + **Alertmanager** | 即時推送、支援多頻道 |
| **部署** | **Docker Compose** (開發) → **Kubernetes** (正式) | 一次配置、可擴展、易於升級 |

> **範例 Docker Compose**（簡易版）  
> ```yaml
> version: "3.8"
> services:
>   db:
>     image: timescale/timescaledb:2.13-pg15
>     environment:
>       POSTGRES_PASSWORD: secret
>       POSTGRES_DB: printer_mgmt
>     ports: ["5432:5432"]
>   snmp-collector:
>     image: ghcr.io/prometheus/snmp_exporter:latest
>     volumes:
>       - ./snmp.yml:/etc/snmp_exporter/snmp.yml
>     ports: ["9116:9116"]
>   grafana:
>     image: grafana/grafana:10.2
>     environment:
>       GF_SECURITY_ADMIN_PASSWORD: admin
>     ports: ["3000:3000"]
>     depends_on: [db]
> ```
> 之後在 Grafana 設定 PostgreSQL 為資料來源，開始建 Panel。

## 8️⃣ 小結

1. **先把印表機硬體、耗材、列印紀錄、故障與網路安全資訊全部抓下來**（SNMP、API、Print Server）。  
2. **以關鍵欄位建表**，特別注意時間序列資料（列印頁數）與耗材百分比。  
3. **定義管理層最關心的 KPI**：利用率、耗材預警、故障率、成本、碳排放、保固剩餘天數等。  
4. **使用 Grafana / Power BI 等工具，將 KPI 以卡片、熱力圖、堆疊條形圖等方式呈現**，並加入即時警示與自動化報表。  
5. **持續收集回饋、優化指標與視覺化**，未來可加入預測模型提升維運效率。
